# 한국식 4구 AI 어떻게 만들었나 — 처음부터 끝까지 쉽게

> 이 글은 강화학습 처음 듣는 사람도 (그래도 Policy Gradient 한 줄은 안다 치고) 따라올 수 있게 썼다.
>
> 한 줄 요약: 컴퓨터가 당구를 못 치다가 → 어느 순간 한 점은 내게 됐고 → 알고리즘은 그대로 두고 "추론할 때 더 똑똑하게 골라보자" 한 것만으로 한 이닝 평균 700점 넘게 침.

---

## 1. 한국식 4구 = 컴퓨터의 놀이터

당구대 위에 공 4개:
- **흰 공 2개**: 내 큐 (CUE_WHITE), 상대 큐 (CUE_YELLOW)
- **빨간 공 2개**: red1, red2

게임 규칙:
1. 내 큐를 쳐서 **빨간 공 두 개를 모두** 맞춰야 1점
2. 도중에 **상대 큐를 건드리면 파울** → 무효
3. **1점 내면 다시 침**. 미스/파울 나오면 차례 끝
4. 한 차례 = "이닝 (inning)" — 이닝 안에 점수 많이 낼수록 잘 친 거

### 컴퓨터에게 게임을 시키려면 뭐가 필요한가

3 가지: 시뮬레이터 + 환경 + 행동 정의.

**(1) 자작 numpy 물리 시뮬레이터**:
- 공 4개의 위치/속도/스핀 추적
- 충돌 (공-공, 공-쿠션) 분석적으로 풀이
- 한 샷 시뮬레이션 비용 ≈ 5 ms (빠름)

**(2) Gymnasium 환경 (`Billiards4BallInningEnv`)**:
- 컴퓨터가 보는 정보 (observation): 4공 각각의 (x, y, vx, vy, ωx, ωy, ωz) = 28 차원
- 컴퓨터가 출력하는 행동 (action) 4 차원:
  - `theta` = 큐가 쳐낼 각도 (0~360°)
  - `power` = 세기 (0~1)
  - `a, b` = 횡 / 종 스핀 (-1~1)
- 보상 (reward): 1점 내면 +1, 파울이면 -페널티, 그 외 0
- 한 이닝 = 미스/파울 / max_shots 까지의 모든 샷

**(3) HTML 시각화**: 학습 후 정책의 trajectory 를 그림으로 보여주는 viewer.

자세한 설계는 `PROJECT_OVERVIEW.md` 참조. 본 글에서는 "환경 다 됐다 — 이제 학습"부터 시작.

## 2. 시작 상태 고정 — PPO vs SAC

### 처음에 한 실험

가장 단순하게: 4공을 **정해진 위치** (canonical) 에 두고 시작 → 컴퓨터가 거기서 1점 내는 정책을 학습할 수 있나?

두 가지 RL 알고리즘으로 시도:

#### PPO (Proximal Policy Optimization)

**정책 기울기 (policy gradient) 의 안전한 친구.**

기본 정책 기울기 알고리즘은 "현재 정책으로 환경에서 데이터 모으고 → 그 데이터로 정책을 더 좋게 업데이트" 인데, 문제는 한 번에 너무 많이 바꾸면 정책이 망가짐. PPO 는:
- 새 정책이 옛 정책에서 **너무 멀리 가지 못하게 잘라냄** (clip)
- "조심조심" 업데이트 → 학습 안정적

쉬운 비유: 산타가 선물 줄 때 "어제보다 좋아진 행동" 만 +α 만큼 더 칭찬. 너무 큰 변화는 무서워서 안 함.

**우리 환경에서는 데이터를 한 번 모아 한 번 쓰고 버림** (on-policy). 효율 안 좋음.

#### SAC (Soft Actor-Critic)

**"적당히 무작위" 도 좋다는 친구.**

기본 정책 기울기 + 두 가지 추가:

1. **모았던 데이터를 다시 씀** (off-policy). 환경에서 한 번 쳐서 모은 (s, a, r, s') 을 큰 통 (replay buffer) 에 저장. 정책 업데이트 할 때 통에서 무작위로 꺼내씀. **데이터 재활용**.

2. **엔트로피 보너스**. 정책이 너무 확정적이면 (= "이 상황엔 무조건 이 행동!") 페널티. 적당히 무작위성을 유지해야 보상 받음. **탐색 잘 함**.

쉬운 비유: 학생이 시험 공부할 때 (1) 옛 문제집 다시 풀기 + (2) "확실한 답" 만 외우지 말고 비슷한 답들도 시도해 보기. 일반화 잘 됨.

#### 학습 환경 (둘 다 동일)

- 환경: `Billiards4BallInningEnv(max_shots=50)`, **canonical 시작** (4공 정해진 위치)
- 학습: 50,000 step
- 시드: 3개 (0, 1, 2)
- 평가: 200 episode 결정적 (deterministic), 같은 canonical 시작

#### 결과

| 알고리즘 | p≥1 (3 seed 평균) | max | mean | 무슨 일이?|
|---|---|---|---|---|
| Random policy | 0.5% | 1 | 0.005 | 운빨 |
| PPO | **0%** | 0 | 0.0 | 한 점도 못 냄 |
| **SAC** | **100%** | **1** | **1.0** | 매번 정확히 1점 |

**SAC 가 PPO 압승**. 이유:
- 환경의 **희소 보상** (점수 내는 샷이 매우 드묾): PPO 는 그 rare 한 샷을 한 번 쓰고 버려서 학습 안 됨. SAC 는 통에 저장해서 계속 재사용.
- **4D 연속 액션 공간** (각도, 세기, 스핀 a, 스핀 b): PPO 의 작은 엔트로피 보너스로는 충분히 탐색 못 함. SAC 의 엔트로피 위주 학습이 잘 cover.

#### 그런데 이상한 점

**SAC 가 매번 정확히 1점, 그 이상 못 냄.** 왜?

답: **시작 위치가 고정** 이라서. SAC 는 "그 정해진 위치에서 점수 나오는 단 한 가지 샷" 만 memorize 함. 200 episode 다 같은 위치 → 같은 샷 → 매번 100% 득점.

그런데 그 샷이 끝난 뒤에 큐 공이 굴러 멈춘 위치는 매번 다름 (랜덤). 그 다른 위치들에서 두 번째 점수 내는 법은 학습 안 됨 → 무조건 미스 → 이닝 종료. **mean = 1, max = 1.**

→ "한 점만 내는 정책" 으로 수렴. Multi-shot 학습 불가능.

## 3. 시작 상태 랜덤 SAC

"한 점만 내는 정책"을 깨려고 다음을 시도: **매 이닝마다 4공 위치 무작위로 배치** → "어디서 시작하든 점수 내는 법" 학습 강제.

### 결과 (50k step SAC, seed 0, 200 episode 평가)

| 평가 모드 | mean | max | p≥1 | foul rate |
|---|---|---|---|---|
| canonical (정해진 시작) | **0.0** | 0 | 0% | 0% |
| random (무작위 시작) | **0.025** | 1 | 2.5% | 19.5% |

**완전 실패.**

해석:
- **canonical 0%**: 랜덤 시작으로 학습한 정책이 정해진 시작에서 한 점도 못 냄. 학습/평가 분포 불일치 (distribution shift).
- **random 2.5%**: 학습 분포에서도 200 ep 중 5개 (2.5%) 만 점수. 거의 random policy 수준 (0.5%).
- **foul rate 19.5% 매우 높음**: 무작위 시작은 큐 공이 종종 상대 공 근처. 학습된 정책이 파울 회피 못 함.

→ Plain SAC 만으로는 안 됨. **환경/보상 설계 자체를 손봐야 함.**

## 4. 병모의 4가지 기여 — 처음으로 multi-shot 등장

협업자 BrianKang-atKAIST 가 4가지를 추가:

### (1) `constrain_aim` — "허공 샷" 막기 ★ 게임 체인저

문제: 정책이 출력하는 각도 θ 가 0~360° 어디든 갈 수 있음. 대부분은 적구를 못 맞춤 (그냥 빈 공간으로 공이 굴러감) → 학습 신호 안 옴.

해법: 정책 출력 θ 를 그대로 안 쓰고, **"가장 가까운 빨간 공 방향 으로 좁은 콘 안" 으로 매핑**:

```
target_dir = atan2(red.y - cue.y, red.x - cue.x)   # cue 가 가장 가까운 red 향한 방향
α         = arcsin(2·공반지름 / cue-red거리)      # 콘의 반각 (이 안이면 무조건 맞음)
offset    = (정책 θ - π) / π                       # [-1, 1] 로 매핑
실제 각도 = target_dir + offset · α                 # 콘 안의 한 점
```

기하학적 의미: arcsin(2r/d) 는 **큐 공 위치에서 본 빨간 공의 "각폭"**. 이 콘 안의 어떤 각도로 쏘아도 첫 접촉이 그 빨간 공에 보장.

→ "빈 공간으로 쏘기" 실패 모드가 환경에서 **불가능해짐**. 모든 샷이 최소 하나의 적구는 맞춤.

쉬운 비유: 정책이 "이쪽 방향" 이라고 가리키면 "여기서 적구가 보이는 각도 안" 으로 자동 조준 보정. 사람이 보조 도와줌.

### (2) `random_start` — 다양한 위치 학습

매 이닝마다 4공 무작위 배치. constrain_aim 과 결합되면 비로소 효과 발휘 — 어느 위치에서든 정책이 콘 안의 각도만 학습하면 첫 접촉 보장됨.

### (3) `gentle_shot` — 득점 후 보너스 (다음 샷 setup)

**득점 샷 직후에만** 발동. 큐 공이 두 번째로 맞춘 빨간 공으로부터 **0.2 m 정도 떨어진 위치**에 멈추면 보너스 peak:

```
bonus = α · exp(-(거리 - 0.2)² / 2σ²),   α = 0.2, σ = 0.1 m
```

당구의 직관: 큐 공이 다음 적구와 너무 가깝지도 (→ 다음 샷 각도 좁아짐) 너무 멀지도 (→ 정밀도 어려움) 않게. ~0.2 m 가 sweet spot.

쉬운 비유: 학생에게 시험만 채점하지 말고, 답안 작성 자세가 좋으면 +α. "다음 문제 풀기 좋은 자세로 멈춰" 학습.

### (4) `foul_penalty` — 파울 페널티 (사실 큰 변화 아닐 수도)

파울 (상대 큐 건드림) 시 음수 보상.

기존 단순 구현:
```
reward = score - foul_penalty   (점수 인정 + 페널티)
```

병모 변경:
```
if fouled:
    reward = -foul_penalty   (점수 무시, 페널티만)
else:
    reward = score + bonuses
```

**근데 simulator 가 이미 `fouled=True → score=0` 보장** 하니까, 둘이 *수학적으로 같음*. 표면적으론 큰 변화 아닐 가능성. 다만:
1. 분기로 보너스 (gentle_shot, setup_shaping) 차단 명확
2. simulator 규칙 변경 가능성 future-proof

**※ 이 변경의 실제 marginal effect 는 검증 필요한 항목** (§11 참조).

### 결과 (SAC 200k step, max_shots=10, continue_on_miss=True)

| 측정 | 값 |
|---|---|
| 평균 (random eval) | **0.575** |
| 최대 inning score | **4** |
| P(≥3) | 14% |
| **무슨 일이?** | **드디어 multi-shot!** Phase 3의 max=1 천장 첫 돌파 |

`constrain_aim` 한 줄이 가장 큰 점프 (mean 0.04 → 0.575, 약 14×). "허공 샷" 실패 모드 제거가 결정적이었음.

## 5. 학습 속도 빠르게 만들기 (Tier A)

학습 속도 측정: n_envs (병렬 환경 수) 를 4 → 8 늘려도 1.1배만 빨라짐. 비정상.

**원인**: 환경의 `info` 딕셔너리에 **trajectory** (100+ 스냅샷) 와 **event log** 가 매 step 통째로 들어 있고, 병렬 worker → main 프로세스로 매 step 이걸 복사함. 학습은 이 데이터를 안 쓰는데도.

**수정**:
1. **A1**: `step()` 이 반환하는 info 에서 무거운 필드 제거 (시각화용은 따로 저장)
2. **A2**: 시뮬레이터 `dt_max` 0.05 → 0.1 (500 episode 검증: 결과 99.4% 일치, 2× 빠름)

**결과**:

| 측정 | Before | After |
|---|---|---|
| n_envs=8 환경 속도 | 599 step/sec | **3034** (5.1×) |
| 200k step 학습 wall | 547초 | **240초** (2.3×) |

함정: n_envs 늘리면 SB3 SAC 의 기본 `gradient_steps=1` 하에서 업데이트 수가 줄어듦 → 품질 하락. `gradient_steps=2` 로 보정해서 회복.

**의미**: 학습 자체는 똑같이 하되 **돌리는 시간 절반** → 같은 시간에 두 배의 실험 가능.

## 6. 우리 contribution: setup_shaping 추가

병모의 `gentle_shot` 은 좋은데, **득점 샷 직후에만 발동**. 학습 초기에는 점수가 거의 안 나오니까 → 보너스도 거의 안 옴 → **신호 sparse**.

### 추가: setup_shaping

**매 비파울 샷마다** 발동 (득점이든 미스든). 큐 공이 가장 가까운 빨간 공으로부터의 거리 d_min 에 대한 지수 감쇠:

```
bonus = α · exp(-d_min / σ),   α = 0.05, σ = 0.3 m
```

d_min = 0 일 때 peak 0.05. 멀어질수록 빠르게 0.

### gentle_shot 과의 차이 4가지

| 측면 | gentle_shot (병모) | setup_shaping (우리) |
|---|---|---|
| 언제 | 득점 샷 직후만 | 매 비파울 샷마다 |
| target | 두 번째 맞춘 red | 가장 가까운 red (어느 거든) |
| 함수 모양 | Gaussian (peak at 0.2 m) | Exponential decay (0 m 일 때 max) |
| 크기 | α=0.2 (큼) | α=0.05 (작음) |

**둘은 동시에 켜져 있음** (alternative 아니라 보완적):
- 미스 → setup_shaping 만 (cue 가 red 근처면 약간 칭찬)
- 득점 → setup_shaping + gentle_shot (점수 1 + 두 보너스)

### 결과 (200k step, 같은 wall)

| 설정 | 평균 |
|---|---|
| 병모 (gentle_shot 만) | 0.575 |
| + setup_shaping (우리) | **0.885** (+54%) |

처음으로 P(≥5) 가 의미있게 등장 (0% → 3.5%).

### 함정 — random_start 와 같이 써야 안전

`setup_shaping` 만 켜고 `random_start` 끄면 (canonical 만 학습) **mean 0.17 로 붕괴**. 

이유: canonical 시작은 어렵 → 정책이 "거의 닿지만 점수 안 내기" 위장 전략에 갇힘. 셰이핑 보너스 (= d_min 작게 만들면 보상) 만 챙기고 점수는 안 냄. **Reward hacking 의 고전적 함정**.

`random_start` 와 함께 쓰면 분포가 다양해서 위장 전략에 못 갇힘.

## 7. 하이퍼파라미터 탐색 (학습 천장 1.17 발견)

밤새 자동 파이프라인으로 여러 hyperparameter sweep:

### 학습 길이

| total steps | 평균 |
|---|---|
| 200k | 0.885 |
| **800k** | **0.975** ✓ sweet spot |
| 1.6M | 0.36 ❌ collapse |
| 2M | 0.44 ❌ collapse |

800k 이상은 셰이핑 신호에 과적합돼서 mean 절반 이하로 떨어짐.

### foul_penalty

| 값 | 평균 |
|---|---|
| 0.5 (병모 default) | 0.975 |
| **0.2** | **1.000** ✓ |
| 0.1 | 0.875 |

### 시드 분산 (같은 config, 7개 seed)

| seed | 평균 |
|---|---|
| 0 | 0.85 |
| 1 | 1.00 |
| 2 | 0.90 |
| 3 | 0.89 |
| **4** | **1.225** ← outlier |
| 5 | 0.98 |
| 6 | 1.01 |

seed 4 가 살짝 운빨이지만 1000-episode 정밀 측정으로 **mean = 1.168 ± 1.78** 확정.

### 시도해봤지만 효과 없는 것 (학습 천장 robust 검증)

| 시도 | 결과 |
|---|---|
| 더 큰 네트워크 [400, 300] | 0.905 ❌ |

→ **학습 천장 = 약 1.17**. SAC + setup_shaping 의 한계.

이 갭은 학습으로 못 줄임 — **다른 접근 필요**.

## 8. Lookahead 기법 — 천장 돌파 시작 ⭐

**여기서부터 알고리즘 / 보상 / 네트워크 안 건드림.** **추론 시점에 더 똑똑하게 고르기** 만 함.

### 발상

학습된 정책의 액션 분포가 너무 spread → 결정적 행동은 그럭저럭이고, 그 주변 stochastic samples 중에 *더 좋은 액션* 이 존재.

**아이디어**: 매 샷마다
1. 정책에서 K개 후보 액션 sample (1개 결정적 + K-1개 무작위)
2. 각 후보를 **시뮬레이터로 1샷 try** (state 복사 + 복원)
3. 가장 점수 높은 액션 실행

학습 안 함. 추론 시간만 더 씀 (K=100 일 때 샷 당 ~0.5초).

쉬운 비유: 학생이 답안 작성하기 전에 머릿속으로 100가지 답 시뮬레이션 → 가장 좋아 보이는 것 적기.

### 1단계 탐색 (h=1) 결과

| K | 평균 | 최대 | P(≥5) | P(≥10) |
|---|---|---|---|---|
| 1 (학습 정책 그대로) | 1.138 | 16 | 6% | 0.6% |
| 5 | 2.55 | 14 | 21% | 4% |
| 10 | 3.71 | 20 | 30% | 10% |
| 20 | 5.45 | 24 | 41% | 21% |
| 50 | 7.69 | 42 | 53% | 30% |
| **100** | **8.62** | **53** | **66%** | **36%** |

K=100 으로 학습 정책 대비 **7.4×**. **학습 한 톨 추가 안 했는데.**

### 2단계 탐색 (h=2)

K개 후보마다 시뮬레이션 후, 그 결과 state 에서 **다시 K2개 후보 평가**. 한 단계 더 깊이 봄.

값 평가:
```
V(a₁) = r₁ + γ · max_{a₂} r₂   (γ = 0.99)
```

| K1 | K2 | sims/shot | 평균 |
|---|---|---|---|
| 100 | 1 | 100 | 8.62 (= h=1) |
| 20 | 5 | 100 | **27.58** (3.2×) |
| **100** | **5** | **600** | **192.0**, max 742 |

**같은 시뮬레이션 수에서 깊이를 늘리면 3배 더 좋아짐.** 깊이 > 폭.

K2 더 크게 (10) 하면 오히려 떨어짐 — 두 번째 layer 가 너무 다양하면 노이즈가 신호를 가림.

## 9. 다중 시드 앙상블 + 적응형 K1 — 최종

### 동기

h=2 mean 192 의 분포 분석: 10 episodes 중 1개가 score=0 (첫 샷 실패). **단일 정책의 sampling 분포 자체에 이 어려운 state 에서의 winning action 이 없음** — K 만 늘려도 안 풀림.

### 다중 시드 앙상블

서로 다른 seed (1, 4, 6) 로 학습된 **3개 정책** 을 동시에 candidate proposer 로 사용.

각 정책이 K_per 개 제안 → 합집합 = 3 × K_per 개 후보. 한 정책이 막힌 상황에서 다른 정책이 다른 "전략 모드" 로 풀 수 있음.

**Note**: 단순 액션 평균 (각 정책 액션 → 평균) 은 망함 (mode 사이 무의미한 중간점). **후보 제안기로 쓰는 게 핵심**. 시뮬레이션으로 검증하고 best 만 선택.

### 적응형 K1 스케줄

샷 인덱스 따라 후보 수 조절:

| shot index | K_per_policy | 총 후보 |
|---|---|---|
| 0 (오프닝) | 200 | 600 (heavy) |
| 1 - 3 | 100 | 300 |
| 4 - 9 | 50 | 150 |
| 10 + | 30 | 90 |

이유: 오프닝에서 실패가 mean 을 가장 깎음 (한 번 미스하면 이닝 끝) → 거기에 가장 많은 compute.

### 결과 (n=10, max_shots=2000)

| 측정 | 값 |
|---|---|
| **평균** | **741.8 ± 646.9** |
| **최대** | **2000샷 (max_shots cap!)** |
| P(≥100) | 80% |
| P(≥500) | **50%** |
| P(≥1000) | **40%** |

10 episodes: [0, 1378, 437, 335, 629, 182, 1406, 1039, **2000**, 12]
- 1개 본질적 실패 (cue 가 가려진 상태)
- 5개가 500점 이상
- 4개가 1000점 이상
- 1개가 2000샷 캡에 막힘

per-shot 정확도 **≥ 99.95%** 추정. 사실상 무한 chain.

### 정리: 학습 정책 → 최종까지

| 단계 | mean |
|---|---|
| 학습 정책 (fast_long_fp02_s4) | 1.17 |
| + K=100 h=1 lookahead | 8.62 (7.4×) |
| + K1=100 K2=5 h=2 lookahead | 192.0 (164×) |
| **+ 다중 시드 + 적응 K** | **741.8 (634×)** |

**Phase 8 이후 모든 향상 (1.17 → 741.8) 은 학습 시간/알고리즘/보상 변화 0** — 순전히 **추론 시점 compute scaling** 만으로.

## 10. PUCT 와의 비교 — 표준 MCTS 가 우리에게 안 통한다

### PUCT 가 뭔가

**PUCT (Predictor + UCT)** = AlphaGo / AlphaZero 가 쓴 표준 트리 탐색.

원리:
1. **Selection**: tree 의 root 에서 leaf 까지 PUCT score 가 가장 높은 자식 선택:
   ```
   PUCT(c) = Q(c) + c_puct · π(c) · √N_parent / (1 + N_c)
              ↑     ↑          ↑      ↑
           평균 보상  정책 prior  탐색  방문수에 반비례
   ```
2. **Expansion**: leaf 에서 K개 후보 sample, 자식 노드로 추가
3. **Evaluation**: leaf 의 즉시 보상 (또는 가치 네트워크)
4. **Backup**: 값을 root 까지 propagate

같은 sim budget 으로 깊은 트리 → 유망한 가지에 더 많이 투자.

**일반적으로 우리 단순 K-expansion 보다 더 좋아야 함**. 더 정교한 알고리즘이니까.

### 결과 (같은 sim budget=600, n=10, max_shots=500)

| 방법 | 평균 | 최대 | P(≥500) |
|---|---|---|---|
| UCT (no policy prior) | 4.4 | 11 | 0% |
| PUCT single (s4) | 86.0 | 156 | 0% |
| **PUCT multi-seed** | **151.5** | 499 | **0%** |
| PUCT multi-seed (2.5× budget=1500) | **146.6** | 500 | 10% |
| **Greedy h=2 multi-seed (우리)** | **357.6** | 500 | **50%** |
| **Greedy h=2 multi-seed (budget 1500)** | **450.0** | 500 | **90%** |

**놀랍게도 우리 단순 방법이 PUCT 를 2.4× 압살**. PUCT 는 budget 늘려도 mean 안 오름.

### 왜 PUCT 가 안 됐나? 세 가지 이유

#### 이유 1: 결정론적 환경에서 visit count 가 무의미

PUCT 의 최종 선택은 "가장 많이 방문한 자식". 그런데 우리 환경은 **결정론적** — 같은 (state, action) 은 항상 같은 결과. **여러 번 방문 = 정보 이득 없음**. UCB의 exploration bonus 가 시간 낭비.

쉬운 비유: 시험 문제의 답이 정해져 있는데 "어디서 자주 봤더라" 통계 내는 격.

#### 이유 2: 연속 액션 공간

PUCT 는 원래 "체스의 합법수 N개" 같은 *유한* 액션 가정. 우리는 4D 연속 → K개 sample 로 근사. 트리 깊이 늘려도 같은 K 안에 갇힘. **Greedy 는 매 노드마다 새 K-sample** 이라 더 자유.

쉬운 비유: PUCT 는 "이 100가지 답 중 골라" 고정. Greedy 는 매번 "100가지 새로 만들어서 골라".

#### 이유 3: Sparse 0/1 보상의 noisy backup

PUCT 의 평균 backup 이 long-horizon return 을 매우 noisy 하게 추정. 잘못된 가지에 budget 집중 가능.

### 의미

**알고리즘이 더 정교한 게 항상 좋은 게 아니다.** 환경 특성에 맞춰야:
- 결정론 + 연속 액션 + sparse reward → 단순 greedy K-expansion 이 정답
- 확률적 + 이산 액션 + dense reward → MCTS/PUCT 가 정답 (AlphaGo)

이게 본 작업의 가장 흥미로운 발견 중 하나.

## 11. 이제 할 것

여러 방향이 있음. 한두 개 골라야 함.

### 옵션 A: 하이퍼파라미터 튜닝 → 성능 한계 측정

현재 mean 741.8 이 max_shots=2000 cap 에 막힌 상태. 진짜 천장이 어디까지?

- K_per_policy 더 키우기 (300, 500)
- h=3 (3단계 lookahead)
- 더 많은 시드 (s1, s4, s6 → 7개 모두)
- max_shots=5000, 10000 으로 cap 해제

예상: mean 1000+, max chain 5000+ 가능 (?)

장점: 빠름 (1-2일).
단점: 정량 개선만, novelty 약함.

### 옵션 B: RLHF 적용 → 사람이 칠 만한 샷

**현재 metric = 점수만**. 그런데 좋은 당구 = 점수 + 스타일.

**Reward Model (RM)** 학습:
- 사람 (또는 GPT-4) 에게 두 inning 비교 → "어느 게 더 좋은 당구"
- 그 선호 데이터로 RM 학습

**Search 시점에 RM 사용**:
```
score = env_reward(action) + λ · RM_score(state, action)
```

RM 학습한 정책 (RL+RM 으로 학습) 대신, **search 시점에만 사용** → reward hacking 위험 적음.

학습 목표:
- 점수 유지 (~700)
- 스타일 향상 (RM 기준)
- 사람 평가 win rate > 60%

장점: ICLR-grade novelty, "search-time RLHF" 패러다임 신선.
단점: 1-2주 작업, 평가 (사람 라벨) 시간 소요.

### 옵션 C: 효과적 시각화 / 발표

지금까지 결과를 어떻게 효과적으로 보여줄지.

후보:
- **인터랙티브 데모**: 사람이 정책 vs 우리 search vs PUCT 를 같은 시작에서 비교, side-by-side replay
- **slider UI**: K, h, multi-seed on/off 슬라이더로 실시간 mean 변화 보기
- **case study 페이지**: 가장 긴 chain (2000샷) HTML + 짧은 chain 비교 → 어떤 상황에서 길어지나
- **paper figures**: matplotlib 으로 K vs mean / h vs mean / budget scaling curve
- **블로그 글**: "AlphaGo 패턴을 4구 당구에 적용한 이야기"

장점: 발표 / 포트폴리오에 강함.
단점: 추가 연구 결과 없음.
